# CLAUDE.md — Mobile App Product Website

## Project Overview

This is a **product/marketing website** for a mobile app. It is built with **Next.js (App Router)**, **Tailwind CSS v4**, and **shadcn/ui** (Radix-based components). The site is bilingual (Finnish as default, English as secondary) and heavily optimized for SEO. It uses a **dark theme** with a modern, techy aesthetic.

> **Hosting is provider-agnostic.** Do not use Vercel-specific features (e.g. `@vercel/analytics`, `@vercel/og`, Vercel Edge Config). Use only standard Node.js / static export compatible solutions.

## Internationalization (i18n)

### Setup

- Use **next-intl** with the App Router plugin.
- Default locale: **`fi`** (Finnish). Secondary locale: **`en`** (English).
- The default locale should **not** have a prefix in the URL (i.e., `example.com/` is Finnish, `example.com/en/` is English). Configure this via `next-intl`'s routing:

```ts
// src/i18n/routing.ts
import { defineRouting } from 'next-intl/routing';

export const routing = defineRouting({
  locales: ['fi', 'en'],
  defaultLocale: 'fi',
  localePrefix: 'as-needed' // No /fi/ prefix for default locale
});
```

### Translation Files

- All UI strings live in `src/i18n/messages/{locale}.json`.
- Structure translations by page/section:

```json
{
  "metadata": {
    "title": "AppName — Kuvaus",
    "description": "Lyhyt kuvaus sovelluksesta..."
  },
  "hero": {
    "headline": "Otsikko tähän",
    "subheadline": "Alaotsikko tähän",
    "cta": "Lataa sovellus"
  },
  "features": { ... },
  "footer": { ... }
}
```

### Rules

- **Never hardcode user-visible strings.** Always use `useTranslations()` or `getTranslations()`.
- Page slugs should be Finnish by default (e.g., `/ominaisuudet`, `/hinnoittelu`). Use `next-intl`'s pathnames config to map English equivalents (`/en/features`, `/en/pricing`).
- Every page must set the `<html lang="">` attribute correctly based on the active locale.

## SEO Requirements

SEO is a **first-class concern** in this project. Follow all of the below rigorously.

### Metadata

- Every page **must** export `generateMetadata()` (or use static `metadata`).
- Include at minimum: `title`, `description`, `openGraph` (title, description, url, images, locale, type), `twitter` (card, title, description, images), `alternates` (canonical + hreflang).
- Use the helper in `src/lib/metadata.ts` to generate consistent metadata across pages:

```ts
// Example: src/lib/metadata.ts
import { Metadata } from 'next';

type GenerateMetadataArgs = {
  title: string;
  description: string;
  path: string;
  locale: string;
};

export function generatePageMetadata({ title, description, path, locale }: GenerateMetadataArgs): Metadata {
  const baseUrl = process.env.NEXT_PUBLIC_SITE_URL!;
  const url = `${baseUrl}${path}`;

  return {
    title,
    description,
    alternates: {
      canonical: url,
      languages: {
        'fi': `${baseUrl}${path}`,
        'en': `${baseUrl}/en${path}`,
      },
    },
    openGraph: {
      title,
      description,
      url,
      locale: locale === 'fi' ? 'fi_FI' : 'en_US',
      alternateLocale: locale === 'fi' ? 'en_US' : 'fi_FI',
      type: 'website',
      siteName: 'AppName',
    },
    twitter: {
      card: 'summary_large_image',
      title,
      description,
    },
  };
}
```

### Hreflang Tags

- Every page **must** include `alternates.languages` pointing to both the `fi` and `en` versions.
- This is critical for Google to correctly index and serve the right language version.

### JSON-LD Structured Data

- Add `SoftwareApplication` schema on the homepage.
- Add `Organization` schema in the root layout.
- Add `BreadcrumbList` schema on all subpages.
- Use a `<JsonLd>` component that renders a `<script type="application/ld+json">` tag:

```tsx
// src/components/seo/JsonLd.tsx
export function JsonLd({ data }: { data: Record<string, unknown> }) {
  return (
    <script
      type="application/ld+json"
      dangerouslySetInnerHTML={{ __html: JSON.stringify(data) }}
    />
  );
}
```

### Sitemap & Robots

- Use **next-sitemap** to auto-generate `sitemap.xml` and `robots.txt` at build time.
- Ensure both locale versions of every page appear in the sitemap with proper `xhtml:link` hreflang annotations.
- `robots.txt` must allow all crawlers and reference the sitemap URL.

### Performance (Core Web Vitals)

- All images must use `<Image>` from `next/image` with explicit `width`/`height` or `fill` + `sizes`.
- Use `next/font` for font loading — no external stylesheet requests.
- Keep JS bundle small: prefer CSS animations over JS. Use `dynamic()` imports for heavy components below the fold.
- Add `loading="lazy"` (default in next/image) for below-fold images; use `priority` for hero/LCP image.
- Target: LCP < 2.5s, CLS < 0.1, INP < 200ms.

### Semantic HTML

- Use proper heading hierarchy (`h1` → `h2` → `h3`). Only one `<h1>` per page.
- Use `<main>`, `<nav>`, `<footer>`, `<section>`, `<article>` appropriately.
- All images must have descriptive `alt` text (in the correct locale).
- All interactive elements must be keyboard-accessible (shadcn/ui handles most of this via Radix).

## Design System & Theme

### Dark Theme

- The site uses a **dark theme by default** with a modern tech aesthetic.
- Set `darkMode: 'class'` in Tailwind config and apply `class="dark"` on `<html>`.
- Define CSS custom properties in `globals.css` following shadcn/ui conventions:

```css
@layer base {
  :root {
    /* Dark theme as default */
    --background: 224 71% 4%;        /* Near-black deep blue */
    --foreground: 213 31% 91%;       /* Light gray text */
    --card: 224 71% 4%;
    --card-foreground: 213 31% 91%;
    --primary: 210 100% 60%;         /* Vibrant blue accent */
    --primary-foreground: 0 0% 100%;
    --secondary: 215 28% 17%;
    --secondary-foreground: 213 31% 91%;
    --muted: 215 28% 17%;
    --muted-foreground: 215 20% 65%;
    --accent: 263 70% 58%;           /* Purple accent for highlights */
    --accent-foreground: 0 0% 100%;
    --border: 215 28% 17%;
    --ring: 210 100% 60%;
    --radius: 0.5rem;
  }
}
```

### Visual Style

- **Gradients**: Use subtle gradient backgrounds (dark blue to near-black). Accent gradients (blue → purple) for CTAs and highlights.
- **Glow effects**: Subtle glowing borders/shadows on cards and interactive elements using `box-shadow` with accent colors.
- **Glass morphism**: Frosted glass effect on cards/nav where appropriate (`backdrop-blur`, semi-transparent backgrounds).
- **Typography**: Clean, modern sans-serif. Use `Inter` or `Geist` via `next/font`.
- **Spacing**: Generous whitespace. Sections should breathe.
- **Animations**: Subtle entrance animations (fade-in + slight translate) on scroll using Framer Motion. Keep it tasteful — no excessive motion.
- **Grid/Bento patterns**: Use grid layouts for feature sections. Asymmetric bento-style grids are encouraged.
- **App screenshots/mockups**: Display in device frames with subtle shadow/glow. Float or tilt for depth.

### Component Conventions

- All shadcn/ui components live in `src/components/ui/`.
- Use the `cn()` utility from `src/lib/utils.ts` for conditional class merging.
- Prefer composition over prop drilling. Build sections as self-contained components.

## Hosting & Deployment

### Provider-Agnostic Rules

- **Do NOT use any Vercel-specific packages** (`@vercel/*`).
- **Do NOT use Vercel Edge Runtime** or middleware features that require Vercel.
- The site must be deployable via `next build && next start` on any Node.js hosting (Hetzner, Railway, Fly.io, AWS, DigitalOcean, self-hosted, etc.).
- Alternatively, if all pages are static, use `output: 'export'` in `next.config.ts` for pure static hosting.
- **Environment variables**: Use `.env.local` for local dev. Document all required env vars:

```bash
NEXT_PUBLIC_SITE_URL=https://example.com   # Used for canonical URLs, sitemap, OG tags
```

### Build

```bash
pnpm install
pnpm build        # next build
pnpm start        # next start (production server)
```

## Code Conventions

- **TypeScript strict mode** — no `any` types unless absolutely necessary and documented.
- **Functional components only** — no class components.
- **Server Components by default** — only add `'use client'` when the component needs interactivity, hooks, or browser APIs.
- **File naming**: kebab-case for files (`hero-section.tsx`), PascalCase for component exports (`HeroSection`).
- **Imports**: Use `@/` path alias for `src/` directory.
- **No barrel exports** (`index.ts` re-exports) — import directly from the component file.
- **Commit messages**: Conventional Commits format (`feat:`, `fix:`, `chore:`, `docs:`).
